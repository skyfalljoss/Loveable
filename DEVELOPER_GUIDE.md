# Developer Guide (Loveable)

## Overview

This project is a modern full-stack Next.js app built with strong end-to-end type safety, a feature-driven modular structure, and scalable data + background job pipelines.

Key technologies:
- Next.js App Router (React Server Components + Client Components)
- TypeScript (strict types across layers)
- tRPC (type-safe API procedures)
- TanStack Query (caching/server state)
- Prisma (DB ORM, migrations)
- Inngest (background jobs/event-driven processing)
- React Hook Form + Zod (form validation + types)
- Clerk (auth + subscriptions)
- Rate Limiter Flexible (usage credit enforcement)
- Shadcn/UI + Tailwind CSS (accessible UI primitives)

This guide explains how each technique is used and why it is beneficial.

---

## 1. Project Architecture (Features + Flow)

High-level flow:
1. User interacts with UI (Client Component)
2. `useTRPC` + `useQuery` / `useMutation` via TanStack Query
3. tRPC router receives request (`/api/trpc` route)
4. tRPC procedure validates with Zod, includes Clerk auth, and calls Prisma/db or Inngest
5. Response goes back to client and Query cache updates
6. Background workflows use Inngest to avoid blocking UI

Benefits:
- End-to-end type safety (DB types propagate to UI automatically)
- Single source for API contracts (no REST endpoint shape drift)
- Solid separation: data fetching server-side, interactivity client-side

## 2. tRPC (Type-safe API layer)

### Why tRPC
- Removes manual request/response schema duplication.
- Autogenerated types across client and server.
- Expressive router model (`queries`, `mutations`).

### Core files
- `src/trpc/init.ts`: `initTRPC`, superjson transformer, base procedure.
- `src/trpc/routers/_app.ts`: app router aggregation.
- `src/app/api/trpc/[trpc]/route.ts`: fetchRequestHandler.
- `src/trpc/client.tsx`: `TRPCProvider` + client setup.

### Context and auth
- `src/trpc/init.ts` imports `auth` from `@clerk/nextjs/server`.
- `createTRPCContext` includes `userId` and clerk session.
- This makes authorization available inside every procedure.

### Router patterns
- Validation with Zod (`.input(z.object(...))`).
- Queries for reads (`.query`), mutations for writes (`.mutation`).
- Example: `messages.getMany`, `projects.create`.

### Invalidation & optimistic flows
- Mutations call `queryClient.invalidateQueries` after success.
- In `message-form`, after `create` we invalidate `messages.getMany` and `usage.status`.

## 3. TanStack Query (Server state + caching)

### Why TanStack Query
- Declarative caching and invalidation.
- Auto refetch and stale/in-memory state.
- SSR hydration (`dehydrate` / `HydrationBoundary`).

### Setup
- `src/trpc/query-client.ts`: `makeQueryClient` with staleTime
- `src/trpc/server.ts`: wrapper to create QueryClient for server.

### Usage styles
- `useSuspenseQuery` for server-bound components (React Suspense handling).
- `useQuery` with fallback for client-only components.
- `useMutation` + `mutationOptions` for safe mutation handling with callbacks.

### Server-side prefetch
- `src/app/projects/[projectId]/page.tsx`: prefetch `messages.getMany` and `projects.getOne` during rendering.
- `dehydrate(queryClient)` into `<HydrationBoundary>` to avoid loading flicker.

---

## 4. Prisma (ORM and DB logic)

### Why Prisma
- Type-safe DB queries.
- Schema-first migrations.
- Auto-generated client + model inference.

### Schema highlights
- `prisma/schema.prisma`: models `Project`, `Message`, `User`, `Usage`.
- `@id` and `@default(uuid())` for IDs.
- `@updatedAt` and `@default(now())` for timestamps.
- Relations: `Project` has many `Message`, `Message` references `Project`.

### Client reuse
- `src/lib/db.ts`: global singleton for PrismaClient in dev to avoid connection exhaustion.

### Common operations
- `findUnique`, `findMany`, `create`, `update`, `delete` implemented in project/procedures.
- Transactional writes for atomic operations (create project + initial message in one call).

## 5. Inngest (Background workflows)

### Why Inngest
- Offloads heavy or slow operations from request path.
- Retries and failure handling built-in.
- Event-driven architecture.

### Key files
- `src/inngest/client.ts`: `new Inngest({ id: "my-app" })`.
- `src/app/api/inngest/route.ts`: `serve(... functions...)`.
- `src/inngest/functions.ts`: define `codeAgentFunction` with steps.
- `src/inngest/functions.ts`: also uses `@inngest/agent-kit` for AI agents (`createAgent`, `createNetwork`, `createTool`, `gemini`).

### Technique: Inngest agent toolkit
- `@inngest/agent-kit` is used for building and orchestrating the AI code agent in functions.
- Contains tools for state/message handling, agent composition and model interaction.

### Typical flow
1. tRPC mutation persists raw user input as one DB record.
2. Sends event: `inngest.send({ name: 'code-agent/run', data: {...}})`.
3. Inngest function executes:
   - Puma sandbox creation
   - AI agent run
   - Save assistant reply as `Message` record
   - (Optional) update project metadata.

### E2B (code interpreter sandbox)
This project includes E2B as a secure code execution sandbox used by the code agent.

- `@e2b/code-interpreter` is installed in `package.json`.
- `src/inngest/functions.ts` imports `{Sandbox}` from `@e2b/code-interpreter`.
- `src/inngest/utils.ts` also imports `{Sandbox}` for shared helper behavior.
- `sandbox-templates/nextjs/e2b.toml` and `e2b.Dockerfile` define runtime template configuration for E2B sandbox environment.

How E2B is used:
1. `Sandbox.create` in Inngest function spins up an isolated container environment.
2. The AI code agent sends generated code to this sandbox for execution and validation.
3. Outputs returned by E2B are incorporated into assistant messages saved via Prisma.

Why E2B:
- Strong isolation for running untrusted code safely.
- Allows the application to evaluate and test code outputs from AI generated content.
- Keeps main app process secure and responsive by offloading code execution.

### Why step.run()
- Step-level retry: if a step fails, only that step re-executes.
- Clear visibility in Inngest logs.

## 6. React Hook Form + Zod (Form UX + validation)

### Why this combo
- Fast uncontrolled form state.
- Minimal rerenders.
- Strong type inference from Zod schema (`z.infer`).

### Example pattern
- Define `formSchema` with `z.object({...})`.
- `useForm({ resolver: zodResolver(formSchema), defaultValues })`.
- `control` prop flows to `FormField` + custom inputs.
- `form.handleSubmit` async triggers server mutation.

### Validation guarantees
- client-side: immediate field errors.
- server-side: tRPC middleware validation enforces shape again.

## 7. Server vs Client Components in Next.js

### Server Components
- Default in App Router.
- Good for data loading and SEO.
- Cannot use state/hooks (useState, useEffect), browser APIs.
- Can directly call async and Perform DB/tRPC prefetch.

### Client Components
- `'use client'` directive.
- Required for event handling, local state, dynamic content.
- Used in: message form, message list, usage component, user controls.

### Hybrid strategy
- Page routes stay server for data fetch,
- interactive modulus in client components.

## 8. File Structure (Feature grouping)

```
src/
  app/                # routes, layouts, providers, API
  trpc/               # transports and routers
  modules/            # feature domains (messages/projects/usage/home)
  inngest/            # background function definitions
  lib/                # helpers (db, usage, utils)
  components/         # shared UI primitives and shadcn wrappers
  generated/          # prisma generated client
```

Why group by feature:
- easier mental model
- smaller modules for single purpose
- no monolithic “api routes” folder logic duplication.

## 9. Authentication + Billing (Clerk + RateLimiter)

### Clerk setup
- `src/app/providers.tsx` wraps app with `<ClerkProvider>`.
- `src/middleware.ts` uses `clerkMiddleware` to protect routes.
- `src/modules/home/ui/components/navbar.tsx` and `user-control.tsx` user session actions.

### Clerk rationale
- OAuth + session management without building authentication flows.
- Built-in multi-factor, social logins.
- Subscription plan and Stripe flows integrated.

### Billing + usage points
- `src/lib/usage.ts`: driver for `RateLimiterPrisma` from `rate-limiter-flexible`.
- Keeps per-user (Clerk userId) usage count in `Usage` table.

`usage.ts`:
- `getUsageTracker` creates `new RateLimiterPrisma({ tableName: 'Usage', points: ...})`.
- `consumeCredits(userId, GENERATION_COST)` decrements points.
- `getUsageStatus` returns remaining points and `msBeforeNext`.

### Using in app flow
- `src/modules/projects/server/procedures.ts` wraps project generation flow with `consumeCredits`.
- On failures (limit exceeded), returns user-friendly message inside assistant response.

### Usage UI
- `src/modules/projects/ui/components/usage.tsx` displays credits and next reset.
- `src/modules/projects/ui/components/message-form.tsx` queries `trpc.usage.status` and conditionally shows usage bar.
- `onSuccess` invalidates usage query to keep data fresh.

### Pricing page
- `src/app/(home)/pricing/page.tsx` uses `PricingTable` from `@clerk/nextjs` and `@clerk/themes`.
- Allows checkout flow built into Clerk/Stripe.

## 10. Common Patterns (with reasons)

- `getQueryClient` + `HydrationBoundary` for SSR from server page
- `useSuspenseQuery` for simpler declarative loading states
- `useMutation(...mutationOptions({ onSuccess: ... }))` to avoid manual cache invalidation code duplication
- `z.object` schemas in tRPC for end-to-end validation
- Secondary generated: `type AppRouter = typeof appRouter` ensures client hooks are type-perfect
- `useAuth` / `useUser` (Clerk) to gate access and redirect unauthorized users.

## 11. Debugging tips

- Use `trpc` devtools in browser to inspect cached query state.
- Raw logs in Inngest functions show per-step events.
- `npx prisma studio` to inspect DB rows for Projects/Messages/Usage.
- Start in dev mode with `npm run dev`; hot reload takes effect for providers and modules.

---

## 12. Contribution workflow

1. Create feature branch.
2. Add tRPC route and procedure in `modules/{feature}/server/procedures.ts`.
3. Add UI in `modules/{feature}/ui/...` and use existing component primitives.
4. Validate with TypeScript (`npm run lint`) and format (`npm run format`).
5. Review data flow from TRPC to Prisma to Query.
6. Add/extend tests if required.
